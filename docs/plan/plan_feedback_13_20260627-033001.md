# 反馈 #13 开发计划：修复思考深度与验收命令修改后被刷新回退

## 背景

反馈 #13 指出：项目配置中的 Codex 思考深度修改无效，验收命令修改无效，修改后会被自动刷新回原值。该问题说明 UI 表单输入、保存 IPC、主进程持久化、状态快照刷新之间存在不同步或覆盖关系。目标是让用户在循环控制/项目配置中修改思考深度和验收命令后，保存结果立即进入持久化状态，并在后续自动刷新、快照轮询、页面切换和计划执行链路中保持一致。

## 设计约束

- 本计划只修复“项目级配置保存后被刷新回退”的根因，不扩展新的 CLI 后端或新的思考深度枚举。
- 验收命令允许为空字符串；空字符串是有效保存值，不能被 `undefined`、旧值、默认值或快照归一化覆盖。
- Codex 思考深度仅在 Codex provider 下生效；选择 Claude 时不能把旧 Codex 深度错误写回或展示为已生效。
- 自动刷新、轮询快照、保存后 reload、跨组件 state 同步都必须以数据库/主进程持久化结果为准，不能用旧闭包状态覆盖用户刚保存的表单值。
- 保存失败必须有明确错误提示或日志，不允许 UI 先显示成功再被刷新回退且无原因。
- 不要把运行测试、构建、回归验证拆成普通开发任务；最终验证统一放到最后“完整验收”节点。

## 总体验收标准

- 配置保存生效：项目级验收命令和 Codex 思考深度修改后能够持久化到主进程/数据库，并被快照读取到最新值。
- 自动刷新不回退：保存后等待自动刷新、页面重新渲染、切换后返回项目，UI 不会恢复为修改前旧值。
- 空命令语义正确：验收命令保存为空字符串时保持为空，并继续表示“不执行外部验收命令”，不会被默认命令或旧命令覆盖。
- 思考深度执行一致：Codex 思考深度保存后，展示、计划生成元数据、任务执行参数中的值一致。
- 错误可见：保存失败时有可见错误或日志，用户输入不被静默回滚。
- 最终验收命令：执行 `npm run check`；验收范围为配置持久化、前端刷新同步、计划/任务读取配置和相关 smoke 覆盖；通过标准为命令退出码 0 且无配置回退相关断言失败。

# 反馈 #5 开发计划：日志卡片粘底滚动优化

## 背景

当前执行日志卡片由 `CodexLog` 展示实时日志或结构化活动行。现有逻辑在 `log` 或 `activity` 更新时直接把日志容器滚动到底部，这能保证用户看到最新日志，但也会在用户主动向上查看前面日志时被新日志打断，导致阅读历史输出不稳定。

反馈 #5 要求：“日志卡片优化一下，如果内容在底部，则添加日志时自动滚动到底部，避免查看前面日志时自动滚动到底部”。本计划默认实现为：日志容器进入“粘底”状态时继续跟随新日志；用户滚动离开底部后暂停自动滚动；用户再次滚动到底部或初次打开日志时恢复自动跟随。

## 任务拆解

- [x] P001: 建立日志底部判定机制 <!-- scope: src/renderer/components/CodexLog.tsx -->
  - 验收要点：在日志滚动容器内定义“接近底部”的判定逻辑，例如允许 16-32px 容差，避免因小数像素或字体渲染导致误判。
  - 验收要点：组件能记录用户当前是否处于底部，不依赖全局状态，不影响 `WorkspacePage` 的运行状态、日志标题和统计信息。
  - 验收要点：用户手动向上滚动超过容差后，组件状态切换为非粘底；用户滚回底部后，状态恢复为粘底。
  - 验收要点：空日志、仅一屏日志、超出一屏日志都能得到稳定判定，不出现滚动高度为 0 时的异常。

- [ ] P002: 实现仅粘底时自动滚动 <!-- scope: src/renderer/components/CodexLog.tsx -->
  - 验收要点：当用户位于底部时，新增日志或活动行后自动滚动到最新内容底部，保持实时跟随体验。
  - 验收要点：当用户正在查看前面日志且不在底部时，新增日志或活动行不会修改 `scrollTop`，当前阅读位置保持稳定。
  - 验收要点：组件初次挂载、从空日志变为有日志、切换到新的日志内容时默认展示底部，避免进入页面后停留在旧内容顶部。
  - 验收要点：自动滚动在 React 渲染完成后执行，避免内容更新前读取旧 `scrollHeight` 造成滚动不到底。

- [ ] P003: 覆盖原始日志与活动行两种模式 <!-- scope: src/renderer/components/CodexLog.tsx -->
  - 验收要点：`activity` 有内容时的结构化活动行模式遵循同一粘底策略，新增活动行不打断用户查看历史活动。
  - 验收要点：`activity` 为空时的原始 `logTail` 模式遵循同一粘底策略，追加原始日志时行为一致。
  - 验收要点：两种模式之间切换时不产生空白、闪跳或错误滚动；当前日志容器仍只渲染一种内容来源。
  - 验收要点：保留现有日志尾部截断展示策略和活动行角色样式，不改变日志文本内容、时间展示和角色标签。

  - 验收要点：手动验证运行中的执行日志：停留底部时新日志持续自动跟随到底部。
  - 验收要点：手动验证向上滚动查看历史日志：后台继续追加日志时页面不跳到底部，阅读位置不被打断。
  - 验收要点：手动验证再次滚动到底部后：后续新增日志恢复自动跟随。
  - 验收要点：`npm run check` 与 `npm run build` 通过，确认 TypeScript、Vite 构建和现有主流程无语法或类型回归。

## 总体验收标准

- 粘底跟随：日志内容当前在底部时，新增日志自动滚动到最新内容底部。
- 阅读保护：用户主动向上查看历史日志时，新增日志不会强制跳到底部。
- 状态恢复：用户重新滚动到底部后，日志卡片恢复自动跟随最新日志。
- 模式一致：结构化活动行和原始日志尾部两种展示模式具备一致的滚动行为。
- 无业务回归：不改变任务执行、停止任务、事件流刷新、日志内容生成、日志截断和页面布局主流程。
- 构建通过：完成实现后 `npm run check` 与 `npm run build` 均通过，关键手动场景验证通过。

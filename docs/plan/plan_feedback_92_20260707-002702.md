# 反馈 #92 开发计划：修复 codex CLI `error: unknown option '--cd'`

## 背景

用户反馈：选择 codex 后端执行时，codex CLI 报错 `error: unknown option '--cd'`，导致计划生成 / 任务执行 / 修复链路在 codex 后端下整体失败。

## 根因分析（基于源码）

- `src/agentCli.js:41-55` 的 `codexNewSessionArgs(workspace, lastFile, options)` 在 `exec` 子命令**之后**注入了 `--cd <workspace>`：
  ```
  exec  -c model_reasoning_effort="..."  --cd <workspace>  --color never  -o <lastFile>  --sandbox danger-full-access  -
  ```
- `--cd`（短形式 `-C`）是 codex CLI 的**全局选项**，按 clap 解析规则必须出现在子命令**之前**（`codex --cd <dir> exec ...`）。放在 `exec` 之后会被 `exec` 子命令的参数解析器接管，部分 codex 版本的 exec 解析器不识别该位置的全局选项，于是抛出 `unknown option '--cd'`。
- 对照 `codexResumeSessionArgs`（`src/agentCli.js:57-67`）：恢复会话路径**从不**传 `--cd`，工作目录完全由 `spawn` 的 `cwd: workspace`（`src/agentCli.js:361`）提供，且该路径长期工作正常。这证明进程 cwd 已足以让 codex 在正确的工作区运行，`--cd` 对 `exec` 是冗余且不兼容的。
- 调用点：`src/loopService.js:1658` `codexNewSessionArgs(workspace, lastFile, { reasoningEffort })`。

## 修复策略

移除 `codexNewSessionArgs` 中的 `--cd` 与 workspace 注入，与 `codexResumeSessionArgs` 对齐，统一依赖 `spawn` 的 `cwd: workspace` 传递工作目录。这样对所有 codex 版本都向后兼容（不再产生 `unknown option`），且工作目录行为与已经稳定的 resume 路径完全一致。

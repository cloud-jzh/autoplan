# 修复计划格式校验因标题漂移误判 plans=0（## 任务拆解 编号前缀规范化）

## 需求概要

反馈 #95：claude 在直写 markdown plan 时把 `## 任务拆解` 写成 `## 2. 任务拆解`，而 validatePlanContent 的正则 /^##\s+任务拆解/m 要求 `##` 后只允许空白再接“任务拆解”，带编号前缀即不匹配，整份 plan 被判为格式不合规、不落库，最终表现为“磁盘有 3 个 plan 文件但 plans=0、无法推进”。planTaskSectionText（/^##\s+任务拆解.*$/m）与 validatePhasePlanContent（校验 总体验收标准 / 进度区）存在同一类脆弱点。修复思路：在计划生成流程中，于落盘/恢复之后、校验之前对 plan markdown 做确定性规范化——把带数字/中文编号前缀或错误层级（###）的 `任务拆解`/`总体验收标准`/`进度区` 标题改写为规范二级标题，补齐任务行缺失的 P0NN 编号与 ` ` 注释，使产物稳定满足 AutoPlan 任务拆解硬性格式；同时强化 markdown 生成 prompt 的标题硬约束，并补充单元测试覆盖规范化与生成兜底。

# P07：Plans、PlanTasks、Acceptance 与 Events 持久化业务状态迁移

## 需求概要

在 P05 Project/Config 写入闭环与 P06 Intake/Attachments/intake_plan_links 完整迁移均有真实通过证据的前提下，依据 D:\docs\autoplan-go-backend-separation-plan.md v1.0，将 plans、plan_tasks、acceptance 与 events 的纯持久化业务状态迁入本机 Go sidecar 模块化单体。先从现有 LoopService、planLifecycle、acceptance、intakeDeletion、snapshot 与测试中冻结 plan/task 合法状态、非法转换、排序、验收/取消验收、批量原子预校验、redo、保护/级联删除、事件顺序和 mutation snapshot；再实现事务型 SQLite repository 与 HTTP/MCP/UI 共用的 application service，支持项目隔离、提交后事件、乐观并发冲突及确定性排序。仅迁移查询、排序、删除、accept/unaccept/redo 等不启动 CLI 或进程的能力；run、stop、resume、re-execute、recreate、run-batches 等长任务只定义版本化契约和能力状态，Go 端稳定返回 NOT_IMPLEMENTED/能力关闭且不得产生状态、事件或进程副作用，HttpAutoplanClient 不得提前把这些动作从 IPC 切走，也不得把失败伪装成 2xx。保持既有 snake_case DTO、字段存在性、null、AppSnapshot、状态机、错误、事件与业务语义，不重写 UI；所有实现和验证只使用脱敏 fixture、系统临时库或调用方显式副本，Go 写入必须持有 owner 锁，Node/sql.js 与 Go 不得同时写同一数据库，且不得自动发现或操作真实 Electron userData。秘密、CLI 配置、会话标识和未授权路径不得进入 API、日志、事件、OpenAPI 示例、fixture 或证据；任一 P05/P06、schema/checksum、owner、显式副本或安全前置未通过时本阶段必须 blocked 并停止。

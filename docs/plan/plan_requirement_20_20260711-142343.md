# P13：Chat 与 MCP 双独立门禁迁移

## 需求概要

在 P10–P12 的 Operation、可恢复 SSE、统一 Process Runner、Files policy、唯一 Go 数据库 owner 与共享 application service 门禁均真实通过后，依据 D:\docs\autoplan-go-backend-separation-plan.md v1.0，以两个互不替代的门禁完成 P13A Chat 和 P13B MCP 迁移。P13A 在 Go 模块化单体中实现 Conversation/Message 持久化、FIFO 队列、停止、部分消息恢复、AI/Claude/Codex provider Operation 以及 chat_chunk/chat_done/chat_queue SSE，并通过独立默认关闭的 Chat feature flag 切换 HttpAutoplanClient；P13B 迁移 localhost HTTP/stdio transport 和现有 MCP 工具目录，使每个工具仅适配 REST/UI 已使用的同一 application service、repository、Files policy、权限、幂等与审计策略，不建立第二套业务规则、仓储或状态机。两个门禁分别测试、分别切换、分别回滚，任一失败只阻止对应通道；旧 Chat 或 MCP IPC 仅在各自门禁稳定证据成立后移除。全过程只使用脱敏 fixture、临时库或调用方明确副本，不自动发现或触碰真实 Electron userData，不允许 Node/sql.js 与 Go 双写，不改变现有 DTO、snake_case、snapshot 与业务规则，并保证失败安全关闭、输出有界且 API key、token、env_vars、正文/tool data 和未授权路径不会进入响应、事件、日志或证据。交付代码、契约文档、自动化测试、真实命令与退出码、脱敏证据、文件清单和剩余风险；前置不满足时标记 blocked，不伪造跨平台或签名结果，不新增 P00 基线失败，且 P13A/P13B 相关测试必须分别通过。

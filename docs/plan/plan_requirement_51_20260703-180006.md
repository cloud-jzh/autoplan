# 需求 #51 开发计划：对话直接创建计划并使用 JSON Schema 约束格式

## 需求概述
对话能力需要支持直接创建执行计划，不再只能通过 `create_requirement` / `create_feedback` 间接触发计划生成。结合当前项目实现，最合理的方案是在 Chat 工具白名单中新增 `create_plan` 工具，让 LLM 通过工具调用提交结构化 plan 数据；工具入参使用 JSON Schema 严格限定计划格式，再由后端统一渲染成 AutoPlan markdown 计划文件、校验格式、写入 `docs/plan`、插入 `plans` 并同步 `plan_tasks`。

## 现状分析
当前 Chat 后端集中在 `src/chat`：`chatController.js` 管理多轮工具调用，`chatTools.js` 注册白名单工具。现有工具包含 `read_file`、`search_files`、`create_requirement`、`create_feedback`、`create_script`、`open_requirement`、`open_feedback`，其中创建需求/反馈会通过 `IntakeService` 间接进入计划生成流程，但没有直接创建 plan 的工具。

计划生命周期能力主要在 `src/loopService.js` 与 `src/loop`：`LoopService.insertPlan()` 复用 `planLifecycle.insertPlan()` 写入 `plans`，`LoopService.syncPlanTasks()` 通过 `planTaskSync.syncPlanTasksFromMarkdown()` 解析 markdown 任务并更新 `plan_tasks`。`src/loop/planGeneration.js` 已有 `isPlanContentValid()`，用于拦截缺少 `## 任务拆解` 或合法任务行的畸形 plan。新增直接创建计划应复用这些生命周期能力，避免 Chat 工具自行拼接散落 SQL。

LLM 工具 schema 当前由 `chatController.formatToolsForProvider()` 转换：OpenAI 兼容路径映射到 `function.parameters`，Anthropic 路径映射到 `input_schema`。因此 `create_plan` 的入参 JSON Schema 可以作为跨 provider 的格式约束基础；若要进一步提高 OpenAI 兼容模型的约束强度，可在工具定义中保留 `strict` / `additionalProperties: false` 等 schema 信息，并确保 `llmClient` 转发时不丢失。

## 方案设计
新增 `create_plan` 工具，工具入参不直接接收 markdown 全文，而接收结构化 JSON：

- `title`：计划标题。
- `summary` / `context`：需求概要或现状分析文本。
- `tasks`：开发任务数组，每项包含 `title`、`scope`、`acceptancePoints`，可选 `details`；后端生成连续 `P001`、`P002`。
- `overallAcceptance`：总体验收标准，包含命令、范围、通过标准。
- `progress`：进度区文本或状态项。
- `status`：可选计划状态，默认 `pending`，必要时支持 `draft`。

后端渲染器负责把结构化 JSON 转成符合 AutoPlan 硬性要求的 markdown：必须包含 `## 任务拆解`，所有任务行独占一行并严格形如 `- [ ] P001: 任务标题 <!-- scope: ... -->`；每个任务后缩进写验收要点但不能写成 checkbox；最后一个任务自动保证为“完整验收”且 `scope: validation`。如果模型已提供完整验收任务，后端规范化；如果未提供，后端追加。工具 handler 写入 `docs/plan/plan_chat_<conversationId>_<timestamp>.md` 或同等稳定命名，调用 `isPlanContentValid()` 和更严格的结构校验后，通过 `LoopService.insertPlan()` / `syncPlanTasks()` 落库。

Chat 工具结果应返回 `{ id, title, filePath, status, totalTasks, type: 'plan', openable: true }`，方便对话工具卡片展示计划创建结果。前端可先以通用工具结果展示，建议补充专门的 plan 卡片，显示计划标题、状态、任务数量和文件路径；若后续已有计划阅读器入口可复用，再接入打开动作。

  - 验收要点：
    - 新增 `create_plan` 工具的 `input_schema`，明确 `title`、`tasks`、`overallAcceptance` 等必填字段和类型，`tasks` 至少包含一个开发任务。
    - schema 对每个任务约束 `title`、`scope`、`acceptancePoints`，禁止任意未声明字段；OpenAI 兼容路径保留严格 schema 所需信息，Anthropic 路径继续使用 `input_schema`。

## 总体验收标准
最终验收命令为：`npm test`、`npm run check`、`npm run build`。

验收范围覆盖 `src/chat/chatTools.js`、新增的 `src/chat/chatPlanTools.js`、`src/chat/chatController.js`、`src/chat/llmClient.js`、`src/main.js`、`src/loopService.js`、`src/renderer/components/workspace/ChatToolCard.tsx`、`src/renderer/types.ts`、`src/renderer/hooks/useChat.ts` 以及新增/更新的测试文件。

通过标准：
1. 对话模型可以通过 `create_plan` 工具直接创建计划，计划文件写入当前项目 `docs/plan`。
2. 计划内容由 JSON Schema 约束的结构化输入渲染，不接受任意散乱 markdown 作为唯一格式来源。
3. 生成的 plan markdown 满足 AutoPlan 任务拆解硬性格式，最后一个任务是完整验收且 scope 为 validation。
4. 成功创建后 `plans` 和 `plan_tasks` 同步更新，对话中返回可识别的 plan 结果卡片。
5. schema 不合规、任务缺少 scope、任务缺少验收要点或最终格式校验失败时，不产生可执行 plan 记录。
6. 现有 Chat 工具、多轮工具调用、队列、标题生成、需求/反馈间接生成计划流程不回归。
7. `npm test`、`npm run check`、`npm run build` 全部通过。

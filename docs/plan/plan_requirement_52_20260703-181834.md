# 需求 #52 开发计划：任务分组排序改为按序号排序

## 需求概述

任务列表按计划分组后，当前排序不应再根据任务执行或更新时间决定，而应按照任务序号排序。结合当前实现，任务序号主要来自 `plan_tasks.sort_order`，由计划 Markdown 中任务行顺序生成；`task_key` 保留 `P001`、`P002` 等可读编号，可作为兼容兜底。

## 现状分析

任务列表展示有两个入口：`src/renderer/components/PlanLists.tsx` 中的工作区任务列表，以及 `src/renderer/components/plans/TaskList.tsx` 中的独立任务列表。二者都调用 `src/renderer/utils/planTasks.ts` 的 `groupTasksByPlan()` 完成分组和排序。

当前 `groupTasksByPlan()` 会先按计划归组，再将组内任务按 `finished_at -> started_at -> updated_at` 的时间降序排列，并把分组按组内最新任务时间降序排列。这会导致后完成或后更新的任务排在 `P001/P002` 等较小序号任务前面，与本需求不一致。

后端解析与快照层已具备序号基础：`src/loop/planTaskSync.js` 从 Markdown 任务行顺序写入 `sort_order`，`src/loop/snapshots.js` 查询任务时已按 `plans.sort_order` 和 `plan_tasks.sort_order` 升序返回。因此本次最优方案是在前端分组工具中去掉时间排序语义，改为序号稳定排序，不改数据库结构和计划解析链路。

## 方案设计

新增或调整任务序号比较逻辑：优先使用有效的 `sort_order` 升序；若历史数据缺失或异常，则从 `task_key` 中解析数字序号作为兜底；仍无法判断时使用原始输入位置保持稳定。组内任务按该比较器排序。分组之间不再使用 `latestTaskTime/sortTime` 抢占顺序，而按分组内最小任务序号、原始分组出现位置稳定排序，避免运行中或最近完成的计划分组跳到前面。

保留状态统计、运行中分组默认展开、筛选计数、任务耗时和更新时间展示等既有行为。验收相关历史列表如仍表达“最近验收”语义，不应被本需求误改；本次只覆盖任务列表分组后的展示排序。

## 总体验收标准

最终验收命令为：在仓库根目录依次运行 `npm test`、`npm run check`、`npm run build`。

验收范围覆盖任务列表分组排序、状态筛选后的排序、任务分组默认展开、两个任务列表展示入口、`groupTasksByPlan()` 单元测试/源代码回归断言，以及与任务列表相邻的 Plan 列表排序行为。

通过标准：
1. `groupTasksByPlan()` 不再使用任务完成时间、开始时间或更新时间作为任务列表分组排序主键。
2. 同一分组内任务严格按 `sort_order` / `task_key` 序号升序展示。
3. 分组之间按分组内最小可见任务序号和原始输入顺序稳定展示，不因最近更新时间变化而跳动。
4. 任务状态筛选、运行中分组默认展开、进度统计、scope 文件展示、任务运行/停止操作保持原有行为。
5. Plan 列表仍保持当前既有排序语义，不被任务列表排序调整误改。
6. `npm test`、`npm run check`、`npm run build` 全部通过，无新增类型错误、构建错误或回归测试失败。

# 需求 #53 开发计划：长需求阶段化生成多个关联计划

## 需求概述

当需求正文较长或明确表达需要分阶段推进时，AutoPlan 不应只生成一个庞大的执行计划，而应拆分为多个阶段性 plan。每个阶段性 plan 都要是标准 AutoPlan 计划文件，能独立进入 `plans` / `plan_tasks` 执行队列，同时这些 plan 必须共同关联回原始需求，便于需求卡片、对话/MCP 查询、删除/中断/恢复等生命周期动作正确处理。

## 现状分析

当前自动计划生成入口在 `src/loop/planGeneration.js` 的 `generatePlanForIntake()`，它为单条 requirement/feedback 计算一个 `planFile`，调用 agent 写入一个 Markdown 文件，随后校验 `## 任务拆解` 与任务行格式，插入一条 `plans` 记录并同步 `plan_tasks`。`src/loopService.js` 的循环扫描只处理 `linked_plan_id IS NULL` 的 intake，一旦生成成功就把 `requirements.linked_plan_id` 或 `feedback.linked_plan_id` 回写为单个 plan id。

需求/反馈的展示与工具详情也建立在单计划关联上：`src/loop/snapshots.js` 通过 `LEFT JOIN plans ON linked_plan_id` 下发 `plan_title/plan_status/plan_completed/plan_total`；`src/renderer/components/IntakePanel.tsx` 只展示一个绑定 Plan 卡片；`src/main.js` 的 `intake:interrupt`、`intake:resume`、`intake:appendTask` 只读取 `linked_plan_id`；`src/loop/intakeDeletion.js` 删除 intake 时也只级联删除一个 plan。要支持阶段化计划，必须补上多计划关联模型，同时保留 `linked_plan_id` 作为兼容字段，避免破坏现有 UI、MCP、Chat 和历史数据。

## 方案设计

新增 `intake_plan_links` 之类的关联表，字段包含 `project_id`、`intake_type`、`intake_id`、`plan_id`、`phase_index`、`phase_title`、`created_at`、`updated_at`，并为既有 `linked_plan_id` 数据做幂等回填。`linked_plan_id` 继续保留为主关联或第一阶段 plan id，所有新逻辑优先读取关联表，缺失时回退单值字段。

长需求判定使用可测试的本地规则：仅 requirement 默认启用，正文长度超过固定阈值、Markdown 标题/列表项数量超过阈值，或正文显式包含“分阶段/多个计划/阶段性计划”等信号时触发；短需求与 feedback 默认仍走单计划路径。触发后，`generatePlanForIntake()` 向 agent 提供明确的多文件输出契约：生成 `plan_requirement_<id>_<timestamp>_phaseNN.md` 多个标准 plan 文件，以及一个 phase manifest，manifest 声明阶段顺序、标题和文件名。后端读取 manifest、逐个校验 plan 内容、插入多条 `plans`、同步各自任务，并通过关联表把所有计划绑定到同一需求；任一阶段文件缺失或格式不合规时，本次生成整体失败，不写入半成品关联。

循环执行继续使用既有 `plans.sort_order` 队列顺序，让多个阶段计划按 phase 顺序连续进入队列。需求完成态回写改为“所有关联计划完成后才把需求标记 completed”；中断/恢复/删除处理所有关联 plan；追加任务默认落到该需求当前首个未完成阶段计划，若都已完成则落到最后一个阶段计划并按现有规则重新激活。

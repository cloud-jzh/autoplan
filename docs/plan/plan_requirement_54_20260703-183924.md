# 需求 #54 开发计划：参考 VS Code 实现执行器模块

## 需求概述

参考 VS Code Tasks 的使用方式，为 AutoPlan 增加独立的“执行器”模块，用于在项目工作区内配置、管理和运行命令型任务。执行器面向构建、测试、脚本命令、项目工具命令等可重复运行场景，不实现 Debug 能力，不接入 `launch.json`、debug adapter、断点、变量检查或调试会话。

## 现状分析

当前项目是 Electron 主进程 + preload + React 渲染层结构。后端已有 `LoopService.runShell()`，可以在项目工作区执行 shell 命令、写入 `docs/progress/logs` 日志、维护 active operation、支持超时和停止；`scripts` 模块已复用这套能力实现手动/钩子/定时脚本。任务队列里的 `tasks` 则是 AutoPlan plan task，由 agent CLI 执行，语义不同于 VS Code Tasks。

执行器模块应独立于现有脚本模块：脚本继续负责 AutoPlan 循环钩子与定时脚本；执行器负责用户可配置的工作区命令入口，参考 VS Code `tasks.json` 的 label/type/command/args/options/group/presentation/dependsOn 等子集。前端需要在工作区左侧“执行”分组中增加入口，并提供列表、编辑、运行、停止、导入 `.vscode/tasks.json` 的界面。

## 方案设计

新增 `src/executors/` 领域模块，包含执行器配置归一化、VS Code Tasks 子集解析、数据库访问和运行编排。数据库新增执行器配置表与必要的最近运行字段；快照下发 `executors` 数组，preload 暴露 `executors:*` IPC。运行时复用 `LoopService.runShell()` 和现有 runtime operation 机制，保证日志、状态和停止行为与脚本/任务执行一致。

VS Code 兼容范围限定为命令执行：支持 `type: "shell" | "process"`、`label`、`command`、`args`、`options.cwd`、`options.env`、`group`、`dependsOn`、`dependsOrder`、`presentation` 的常用字段；`problemMatcher` 可保存和展示为配置元数据，但首版不实现完整诊断系统。明确拒绝或忽略 debug 专属配置，`.vscode/launch.json` 不导入、不展示、不执行。

# 需求 #55 开发计划：增加终端模块，参考 VS Code 集成终端

## 需求概述

为 AutoPlan 增加软件内部的终端模块，让用户可以在工作区内创建、管理和使用交互式命令终端。该模块参考 VS Code Integrated Terminal 的核心体验：项目级终端入口、多终端会话、实时输出、键盘输入、窗口尺寸同步、重命名、清屏、停止/删除会话、默认工作目录和 shell/profile 管理。

本需求不等同于现有脚本模块或执行器模块。脚本/执行器偏向可配置的一次性命令运行与日志回写；终端模块偏向长期存在的交互式 PTY 会话。终端输出应通过事件流式推送到渲染层，不应把高频 stdout/stderr 数据塞入现有 `snapshot`。

## 现状分析

当前项目是 Electron 主进程 + preload + React/Vite 渲染层结构。`src/main.js` 集中注册 IPC，`src/preload.js` 通过 `contextBridge.exposeInMainWorld('autoplan', ...)` 暴露安全 API，`src/renderer/pages/WorkspacePage.tsx` 以单页 tab 承载工作区功能，`src/renderer/types.ts` 集中定义 `WorkspaceTab`、`AppSnapshot` 与 `AutoplanApi` 类型。

现有 `LoopService.runShell()` 能运行一次性 shell 命令、写入 `docs/progress/logs`、维护 `runtime.activeOperations` 并支持停止；但它不适合交互终端，因为它基于 child_process + shell command，面向命令执行结果而非长期 PTY 会话，也没有输入、resize、滚动缓冲和多会话生命周期。当前仓库已有 `src/executors/` 的执行器配置/存储雏形，但尚未完整接入主进程、preload、快照和 UI。终端计划应预留与执行器联动的入口，但不依赖执行器先完成。

## 方案设计

新增 `src/terminal/` 领域模块，由主进程持有终端会话池。优先采用 `node-pty` 提供真实伪终端能力，渲染层采用 xterm.js 系列包渲染终端。主进程负责校验 `projectId`、解析工作区路径、选择 shell/profile、创建 PTY、写入输入、调整 cols/rows、终止进程、维护 session 元数据与有限 scrollback；渲染层只通过 preload API 订阅数据和发送用户输入，不暴露 Node/Electron 对象。

会话元数据可进入 `AppSnapshot` 或独立 `terminal:list` 返回，用于导航徽标和会话列表；实时输出使用 `terminal:data`、`terminal:exit`、`terminal:status` 等事件推送。终端默认 cwd 为当前项目 `workspace_path`，允许在创建时选择工作区内子目录；对越界 cwd 拒绝创建。终端内部用户自行输入的 `cd` 行为由 shell 负责，不尝试拦截，但 UI 需要明确展示当前创建时 cwd，并对非工作区 cwd 创建请求做边界保护。

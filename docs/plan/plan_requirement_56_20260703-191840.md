# 需求 #56 开发计划：需求与反馈显示使用的 CLI

## 需求描述

> 需求和反馈，要显示使用的 cli

在「需求」和「反馈」两个工作区视图里，每条需求 / 反馈记录卡片上需要展示该条记录所使用的 CLI 后端（Codex / Claude / OpenCode / Oh My Pi），让用户一眼看到每条记录由哪个 CLI 负责生成计划与执行任务。

## 背景与现状分析（基于源码）

- 数据层已就绪：`requirements` / `feedback` 两张表（`src/database.js`）已含 `agent_cli_provider`、`agent_cli_command`、`codex_reasoning_effort` 列；`src/renderer/types.ts` 的 `Requirement` / `Feedback` 接口也已声明这些字段。
- 创建时已写入：`src/intakeService.js` 的 `createRequirement` / `createFeedback` 通过 `nextIntakeAgentCliConfig` 把 CLI 配置持久化到记录上；UI 侧 `WorkspacePage.tsx` 用 `ComposerCliSelectionProvider` 在发送需求 / 反馈时携带所选 CLI。
- 快照已下发：`src/loop/snapshots.js` 以 `SELECT requirements.*` / `SELECT feedback.*` 取出全部列，并用 `intakeSnapshotRow`（来自 `src/loop/agentCliConfig.js`）规范化 `agent_cli_provider` 等字段后下发给渲染层。
- 缺口 1（数据准确性）：当需求 / 反馈创建时未显式指定 CLI 时，记录上的 `agent_cli_provider` 为 `null`，此时真正“使用的 CLI”应为项目（`project_states`）默认 CLI。已存在工具函数 `effectiveAgentCliConfig(defaults, override)`（`src/loop/agentCliConfig.js`，已 export）可把“记录覆盖优先，否则回退项目默认”解析为生效 CLI，但快照当前未对 intake 行做这一步解析。
- 缺口 2（展示）：`src/renderer/components/IntakePanel.tsx` 渲染需求 / 反馈卡片（标题、状态 chip、正文、附件、绑定 Plan 卡片、操作区、更新时间），但**完全没有展示 CLI**。
- 可复用资产：`src/renderer/components/shared.tsx` 已提供 `planCliSummaryLabel(source)`（返回如 `Codex CLI · 思考深度 medium` / `Claude CLI` / `OpenCode CLI` / `Oh My Pi CLI`）、`agentCliProviderLabel()`、`readAgentCliProvider()`；样式上已有 `.cli-tag`（计划卡片 `PlanList.tsx` 在用）和 `.project-cli-badge`（`ProjectsPage.tsx` 在用）两种现成视觉模式。

结论：本需求以**纯展示**为主，无需改动创建 / 编辑流程；核心是 (1) 在快照层把每条 intake 的“生效 CLI”解析出来，(2) 在 `IntakePanel` 卡片上用既有标签样式渲染出来。

## 总体方案

1. 快照层补全生效 CLI：在 `snapshots.js` 构建 `requirements` / `feedback` 行时，用 `effectiveAgentCliConfig(state, intakeRow)` 把记录覆盖 + 项目默认解析成生效 provider / effort，写入快照行（沿用 `intakeSnapshotRow` 已有的“就地规范化覆盖”惯例，覆盖 `agent_cli_provider` / `agent_cli_command` / `codex_reasoning_effort` 字段）。这样渲染层只管显示，逻辑单一来源。
2. 渲染层加 CLI 标签：`IntakePanel.tsx` 每张卡片标题区（状态 chip 旁）渲染一个 `.cli-tag`，文案取 `planCliSummaryLabel(item)`；编辑态标题区同步展示。
3. 复用 / 微调 `.cli-tag` 样式，保证在 intake 卡片标题区与状态 chip 并排时不溢出 / 不换行错位。
4. 补测试覆盖生效解析与标签渲染。

> 本计划只做“显示”，不改动需求 / 反馈的创建与编辑表单（编辑表单是否可改 CLI 不在本需求范围内）。

- 质量：`npm test` 通过、`npm run build` 通过。

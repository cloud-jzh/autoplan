# 需求 #57 开发计划：计划生成失败原因与可切换 CLI 重试

## 需求描述

> 需求和反馈，生成计划失败时，要在卡片显示失败原因，要支持重试，要支持切换cli进行重试

## 现状分析

- 自动循环在 `src/loopService.js` 的 `runOnce()` 中扫描未绑定 Plan 的需求和反馈；生成失败后依赖 `generate_fail_count < 3` 或 15 分钟冷却再次尝试，目前没有用户主动重试入口。
- `src/loop/planGeneration.js` 的 `generatePlanForIntake()` 已在 CLI 退出失败、未写入计划文件、Plan 格式不合规时记录事件并自增 `generate_fail_count`，但只持久化失败次数和时间，没有持久化可展示的失败原因。
- `src/database.js` 现有 `requirements` / `feedback` 表包含 `generate_fail_count`、`last_generate_fail_at`，缺少 `last_generate_error`、`last_generate_log_file` 等卡片直读字段。
- `src/loop/snapshots.js` 通过 `SELECT *` 下发需求/反馈快照，并在 `intakeLinkedPlanSnapshotRow()` 中解析生效 CLI；新增失败字段后可在同一快照链路暴露给渲染层。
- `src/renderer/components/IntakePanel.tsx` 当前只在已有 `linked_plan_id` 时展示绑定 Plan 卡片；未生成 Plan 且失败的需求/反馈卡片看不到失败原因，也没有“重试生成计划”操作。
- 创建需求/反馈时的 CLI 选择已在 `Composer.tsx`、`workspaceForms.ts`、`nextIntakeAgentCliConfig()` 中有现成模型，可复用于失败卡片上的“切换 CLI 后重试”。

## 总体方案

新增“计划生成失败状态”作为需求/反馈记录的一部分：生成失败时写入失败原因、日志路径、失败时使用的 CLI；生成成功或手动重试前清理失败状态。新增 `intake:retryGeneratePlan` IPC：允许对未绑定 Plan 的需求/反馈重置失败节流、可选覆盖 CLI 配置，并触发一次生成尝试。前端在需求/反馈卡片中检测失败状态，展示错误原因、失败时间、日志路径和重试控件；重试控件支持切换 Codex / Claude / OpenCode / Oh My Pi，Codex 下支持选择思考深度。

## 总体验收标准

- 功能标准：需求卡片和反馈卡片在计划生成失败后均能直接显示失败原因、失败时间、失败次数和日志路径；用户可在卡片上切换 CLI 并重试生成计划。
- 数据标准：失败状态持久化到对应 intake 记录；生成成功或重试前按预期清理失败状态；CLI 覆盖只影响当前 intake，不污染项目全局配置。
- 流程标准：自动循环原有失败节流继续生效；手动重试能显式绕过失败冷却并触发一次生成尝试；忙碌中的项目不会启动并发生成。
- 兼容标准：现有需求/反馈创建、编辑、删除、附件、绑定 Plan 预览、中断/恢复、追加任务、CLI 默认配置、事件流不回归。
- 最终验收命令：`npm test`、`npm run build`。
- 通过标准：两条命令均成功；手工构造失败和重试场景通过；无新增控制台异常或明显布局溢出。

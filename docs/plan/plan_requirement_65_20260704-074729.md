# 需求 #65 开发计划：大数据量工作区性能优化

## 需求概述

在 requirements、feedback、plans、tasks、attachments、scan_files 数据量增长后，工作区需要减少主进程同步阻塞、全量 IPC payload、渲染进程派生计算和一次性 DOM 渲染。目标是在不替换 sql.js、不重构全部 IPC 协议、不改变业务语义的前提下，优先解决 P0 的 snapshot N+1 和 scan_files 全量传输，再推进前端渲染、派生计算、刷新策略和同步 I/O 降噪。

## 现状分析

1. `src/loop/snapshots.js` 的 `snapshot()` 当前一次性组装项目、状态、requirements、feedback、attachments、plans、tasks、events、scan_files、scripts、executors、terminals 和运行态。
2. requirements/feedback 在 `snapshot()` 中逐行调用 `linkedPlanSnapshotsForIntake()`，后者进入 `src/loop/intakePlanLinks.js#getPlansForIntake()`，对大量 intake 形成 N+1 查询。
3. `snapshot()` 直接执行 `SELECT * FROM scan_files WHERE project_id = ? ORDER BY scanned_at DESC, file_path ASC`，常规 `loop:update` 会把全量扫描文件带到渲染进程。
4. `src/chat/chatTools.js` 的文件搜索/列表能力直接查询 `scan_files`，工作区 UI 当前没有直接展示全量 `snapshot.scans`，因此可先把常规 snapshot 与后端 scan_files 能力解耦。
5. `src/loop/runtime.js#createThrottledUpdateEmitter()` 当前每次刷新都会调用完整 `snapshot(projectId)` 并通过 `loop:update` 推送。
6. `src/renderer/hooks/useSnapshot.ts` 只处理完整 `AppSnapshot`，有 requestAnimationFrame 合并，但没有增量合并。
7. `src/renderer/hooks/useWorkspaceController.ts` 多个 `useMemo` 依赖整个 snapshot 或大数组；搜索虽在空 query 时返回空结果，但 controller 仍会随 snapshot 引用变化触发后续过滤、任务标题映射和验收分组计算。
8. `src/renderer/components/IntakePanel.tsx` 对所有 `items` 直接 `map`，且每条记录渲染时执行 `attachments.filter(...)`。
9. `src/database.js` 基于 sql.js，每次 `run`/`insert`/`runBatch` 持久化都会 `db.export()` 并写回整个文件；应减少无意义写入和由此引发的全量刷新。
10. `src/loop/workspaceFiles.js` 仍存在同步扫描、同步 stat/hash/readFile 路径；目前已有 `scanDirectoryInWorker()`，计划中应避免 UI 高频刷新路径触发同步文件 I/O。

## 设计原则

- 保持 `AppSnapshot` 现有字段的向后兼容；移除大 payload 时优先保留空数组、摘要或按需接口，不让老调用方崩溃。
- P0 先降低主进程 snapshot 查询量和 IPC payload；P1 再减少渲染计算与 DOM 数量。
- 不改变需求/反馈/计划/任务/扫描记录的数据结构语义，不删除历史数据。
- 新增测试以性能路径为主，覆盖查询次数、payload 规模、空搜索短路、大量 intake 首屏渲染和增量刷新合并。

# 需求 #66 开发计划：工作区终端页接入完整交互终端

## 需求概述

当前工作区「终端」页点击「新建」后看起来无效，根因不是后端终端能力缺失，而是 `src/renderer/pages/WorkspacePage.tsx` 的 terminal tab 仍渲染 `WorkspaceTerminalMetadataSection`。该组件只展示会话元数据列表，并在页面层直接调用 `window.autoplan.createTerminal({ projectId })`，既没有挂载 xterm.js 屏幕，也没有复用 `useTerminalSessions` 对 `{ ok: false }` 的错误展示。

本次目标是把 terminal tab 切到既有完整终端组件 `WorkspaceTerminalView`：创建后能看到 xterm 面板、PTY 输出和会话 tab，支持输入、resize、replay、stop、clear、rename，并保留页面级会话刷新逻辑用于侧栏活动数量和首次列表兜底。

## 现状分析

1. `WorkspacePage.tsx` 已维护 `terminalSessions`、`terminalListLoaded`、`refreshTerminalSessions`，并监听 `onTerminalStatus` / `onTerminalExit`，`currentTerminalSessions` 会按 `projectId` 过滤后供侧栏活动徽标统计。
2. `WorkspacePage.tsx` 的 terminal section 当前渲染 `WorkspaceTerminalMetadataSection`，它只显示状态、cwd、shell/尺寸/退出信息，不会创建 xterm 实例，也不会订阅 terminal data。
3. `WorkspaceTerminalView.tsx` 已提供完整交互终端 UI，props 包括 `projectId`、`terminals`、`scripts`、`executors`、`packageScripts`、`workspacePath`、`autoRefresh`，内部通过 `useTerminalSessions({ projectId, initialSessions: terminals, autoRefresh })` 管理会话。
4. `WorkspaceTerminalView.tsx` 的 `TerminalPane` 已创建 `XTerm` + `FitAddon`，把 `xterm.onData` 写回 PTY，会订阅后端 data、resize、replay，并提供新建、清屏、重连、停止、重命名和快捷命令入口。
5. `useTerminalSessions.ts` 已在 create/list/write/resize/kill/rename/clear/replay 等路径检查 `result.ok`，失败时写入 `error`，`WorkspaceTerminalView` 会用 `.terminal-error` 展示该错误。
6. `AppSnapshot.activeProject` 和 `Project.workspace_path` 已存在，可作为 `WorkspaceTerminalView.workspacePath` 的主来源；路由项目 `routeProject` 可作为兼容兜底。
7. `WorkspacePage.test.tsx` 现有终端回归仍断言 metadata section 展示，需要改为断言页面接入 `WorkspaceTerminalView` 并传递当前项目的 terminals/scripts/executors/workspacePath。

## 设计原则

- 不改后端 IPC、preload、TerminalService、node-pty 或 xterm 组件能力，优先修正页面接线。
- 页面级终端列表仍负责侧栏活动数量、首次快照兜底和 status/exit 事件的 project-scope 同步。
- 创建失败、PTY 不可用、workspace/cwd 无效等错误交给 `WorkspaceTerminalView` + `useTerminalSessions` 展示，避免页面层 metadata 按钮吞掉 `{ ok: false }`。
- 删除不再使用的 metadata-only 组件和展示辅助函数，避免继续保留一个不可交互的伪终端页。
- 测试优先覆盖页面接入契约，确保以后 terminal tab 不会退回元数据列表。

# 需求 #67 开发计划：计划生成与任务执行后端解耦

## 需求概要

当前 AutoPlan 的计划生成、任务执行、验收修复共享旧的 `agentCliProvider` / `agentCliCommand` / `codexReasoningEffort` 配置。计划生成阶段由外置 CLI 直接写 Markdown，格式稳定性依赖 prompt；任务执行阶段又通过 `planAgentCliConfig` 从 plan、事件或 intake 反推同一套 CLI 配置，导致“内置 LLM 稳定生成计划 + 外置 CLI 执行任务”等组合无法独立配置。

本次目标是新增计划生成配置与计划执行配置两组独立概念，并引入结构化中间格式 `PlanSpec`。`external-cli-markdown` 保持现有行为作为默认兼容策略；`external-cli-structured` 与 `builtin-llm-structured` 统一产出 `PlanSpec`，再由 AutoPlan 校验并确定性渲染为现有 `syncPlanTasks` 可解析的 Markdown。执行侧第一阶段继续复用外置 CLI 路径，`builtin-llm` 执行只保留配置并返回明确未支持错误。

## 现状分析

1. 旧配置归一化集中在 `src/loop/agentCliConfig.js`，`effectiveAgentCliConfig` 已负责项目默认值、intake 覆盖和旧字段兼容。
2. 计划生成入口在 `src/loop/planGeneration.js`：普通 issue scan 和 requirement/feedback intake 都通过 `service.runCodex` 调外置 CLI 直接写 Markdown；intake 路径已具备 `validatePlanContent`、`validatePlanTaskSequence`、stdout 兜底、失败计数和 `on:fail` 钩子链路。
3. 任务执行和验收修复通过 `src/loop/taskExecution.js`、`src/loop/validation.js` 调 `service.planAgentCliConfig(plan)`，再转为 `runCodexWithPlanGuard -> runCodex -> agentCli.js spawn`。
4. `src/loop/planAgentCli.js` 对老 plan 通过 plan 行、`plan.generated` 事件、requirements/feedback 来源和项目默认值回退，这正是兼容旧快照需要保留的逻辑。
5. 数据库当前 `project_states`、`requirements`、`feedback`、`plans` 只有旧 `agent_cli_*` 与 `codex_reasoning_effort` 字段；`plans` 插入由 `src/loop/planLifecycle.js` 的 `insertPlan` 写入旧 CLI 快照。
6. 内置 LLM 基础能力已存在于 `src/chat/llmClient.js` 与 `src/chat/aiConfigService.js`，支持 OpenAI 兼容、DeepSeek、Anthropic、工具定义和思考参数，可作为结构化计划生成的底层客户端。
7. MCP 工具 schema 在 `src/mcpTools.js`，工具文档由 `src/mcpToolDocs.js` 根据 schema 派生；现仅暴露旧 `agentCli*` 字段。
8. UI 设置页 `WorkspaceSettingsView.tsx` 当前只有一个“CLI 后端”配置区；Composer 只允许为单条需求/反馈选择旧 `agentCliProvider` 与 Codex reasoning。

## 设计原则

- 默认行为必须兼容：未配置新字段时仍走 `external-cli-markdown` + `external-cli`，并继续使用旧 `agentCli*` 默认值。
- 生成配置与执行配置独立：生成阶段只读取 `planGeneration*`，执行阶段只读取 `planExecution*`，执行阶段不得误用 `planGenerationProvider`。
- 结构化路径模型只生成 `PlanSpec`，不直接生成 Markdown、不负责任务编号；AutoPlan 负责 schema 校验、规范化、编号、完整验收任务和 Markdown 渲染。
- 渲染后的 Markdown 必须继续复用现有 `validatePlanContent`、`validatePlanTaskSequence` 和 `syncPlanTasksFromMarkdown`，不引入第二套任务解析格式。
- `builtin-llm` 执行第一阶段不静默回退，必须明确失败，避免用户误以为内置执行已具备文件读写、权限和恢复能力。

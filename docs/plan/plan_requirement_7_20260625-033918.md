# 需求 #7 开发计划：同一 Plan 任务持续使用同一上下文

## 背景

当前 AutoPlan 在执行 plan 任务时，每次调用 Codex 都会通过 `codex exec` 启动一个新的非交互式会话。对于同一个 plan 任务，如果因为失败、停止、重试或手动再次执行而需要继续推进，新的会话无法继承该任务上一次的理解、已做分析和执行上下文，容易造成重复探索、遗漏前序约束或与上一轮执行结论不一致。

需求 #7 要求：同一个 plan 任务应该在同一个上下文内部持续执行。实现目标是以 `plan_id + task_key` 为任务上下文边界，首次执行创建 Codex 会话，后续同一任务继续执行时恢复该会话；不同任务仍保持隔离，避免上下文串扰。

## 任务拆解

- [x] P001: 持久化任务上下文标识 <!-- scope: src/database.js,src/renderer/types.ts -->
  - 验收要点：`plan_tasks` 表新增可空的 Codex 会话标识字段，能够记录同一任务首次执行产生的 session id。
  - 验收要点：新增字段通过 `ensureColumn` 兼容历史 SQLite 数据库，旧任务缺失会话标识时不影响启动、快照和任务列表展示。
  - 验收要点：前端 `PlanTask` 类型同步新增上下文字段，现有任务耗时、状态、scope、排序字段类型不回退。
  - 验收要点：会话标识仅绑定到具体 `plan_tasks` 记录或 `plan_id + task_key`，不使用全局单例存储，避免跨 plan 或跨任务复用。

- [x] P002: 保留任务同步中的上下文 <!-- scope: src/loopService.js -->
  - 验收要点：`syncPlanTasks` 在重新解析 plan 文件时保留同一 `plan_id + task_key` 已有任务的 Codex 会话标识、耗时字段和关键状态信息。
  - 验收要点：任务标题、raw line、scope 或排序变化时仍能更新最新解析结果，但不会因为删除重建任务而丢失上下文。
  - 验收要点：任务 key 被移除或改名时不把旧会话错误迁移到新任务；新增任务默认没有会话标识。
  - 验收要点：已完成任务的 checkbox 与进度区由现有 AutoPlan 回写流程继续负责，上下文保留逻辑不扩大 plan 文件写入范围。

- [x] P003: 支持 Codex 会话恢复执行 <!-- scope: src/loopService.js -->
  - 验收要点：`runCodex` 支持传入已有 session id；存在 session id 时使用 `codex exec resume <session_id> -` 继续会话，不再为同一任务创建全新上下文。
  - 验收要点：没有 session id、session id 为空或恢复失败时能够回退为新建 `codex exec`，并记录可排查事件，不导致任务流程卡死。
  - 验收要点：新建会话时继续保留当前 `--cd`、`--color never`、`--output-last-message`、沙箱和 stdin prompt 行为；恢复会话时不传入 Codex CLI 不支持的参数。
  - 验收要点：从 Codex 输出中捕获真实 session id，写入运行结果，日志文件和 last message 文件路径仍按现有规则生成。

- [x] P004: 绑定任务执行与上下文生命周期 <!-- scope: src/loopService.js -->
  - 验收要点：`executeTask` 执行前读取当前任务保存的 session id，并把它传给 Codex 调用；执行结束后把捕获到的 session id 写回对应任务。
  - 验收要点：同一任务失败后回到 `pending`、被停止后再次执行、或用户手动点击运行时，都会复用该任务已有上下文。
  - 验收要点：不同任务即使属于同一 plan，也不会复用彼此的 session id；并发任务各自维护独立上下文。
  - 验收要点：任务成功完成后保留 session id 作为审计信息，但不会让后续其他任务自动继承该上下文。

- [x] P005: 暴露上下文状态与事件信息 <!-- scope: src/loopService.js,src/renderer/types.ts,src/renderer/pages/WorkspacePage.tsx,src/renderer/components/CodexLog.tsx -->

  - 验收要点：事件流在任务开始、恢复、回退新建、失败和完成时包含可读的上下文信息，方便定位某次任务是否复用了上下文。
  - 验收要点：工作区 Codex 日志区域能展示当前执行是新会话还是恢复会话，不影响原有 logTail 和 activity 展示。
  - 验收要点：界面只显示必要的短标识或状态文案，避免长 UUID 造成布局溢出；无 session id 的历史任务显示为普通任务。

## 总体验收标准

- 上下文连续：同一个 plan 任务在失败、停止、重试或手动再次运行时，优先恢复该任务上一次的 Codex 会话继续执行。
- 任务隔离：上下文以 `plan_id + task_key` 为边界，不同任务、不同 plan、不同项目之间不会共享会话。
- 数据可靠：会话标识持久化到 SQLite，并在 plan 文件重新同步时保留；历史数据和无会话任务保持兼容。
- 流程稳定：plan guard、任务状态流转、并发 scope 控制、日志输出、进度区回写和验收流程保持原有行为。
- 可观测：事件流、活动日志或任务快照能够判断本次执行是新建上下文还是恢复上下文，并能定位关联 session。
- 验证通过：完成实现后 `npm run check` 与 `npm run smoke` 均通过，必要时补充手动验证记录。
